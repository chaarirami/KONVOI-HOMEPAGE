Per FND-07 and FND-08: Replace the entire `.github/workflows/actions.yaml` file with:

```yaml
name: CI

on:
  pull_request:
    branches:
      - main
  push:
    branches:
      - main

jobs:
  build-and-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 10

      - name: Use Node.js 22
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Type-check
        run: pnpm run check

      - name: Build
        run: pnpm run build

      - name: Check for AstroWind debris in dist/
        run: |
          if grep -riq "astrowind\|arthelokyo\|onwidget\|Unsplash\|Cupertino" \
               --include="*.html" --include="*.js" --include="*.css" dist/; then
            echo "ERROR: AstroWind debris detected in dist/"
            exit 1
          fi
          echo "Grep gate passed"
```

This replaces the current workflow which has these problems:
- Uses `npm ci` instead of `pnpm install --frozen-lockfile`
- Uses `cache: npm` instead of `cache: pnpm`
- Tests Node 18, 20, 22 in a matrix (project requires Node 22+)
- Has two separate jobs (build + check) instead of one

Key design decisions baked into the replacement:
- `pnpm/action-setup@v4` with `version: 10` (official pnpm GitHub Action, matches project)
- `cache: pnpm` in setup-node caches the pnpm store keyed on pnpm-lock.yaml
- `--frozen-lockfile` prevents silent lockfile mutations in CI
- Single job replaces two-job structure (simpler, saves runner minutes)
- Node 22 only (project engines require `^22.0.0 || >=24.0.0`)
- Grep gate runs after build in the same job -- case-insensitive (`-i`), targets only `*.html`, `*.js`, `*.css` files in `dist/`
- The `astrowind:config` virtual module ID in `vendor/integration/` does NOT appear in `dist/` output (Vite resolves it at build time), so no false positives expected

**netlify.toml (FND-10):**

Replace the entire file with:

```toml
[build]
  publish = "dist"
  command = "pnpm run build"

[build.processing.html]
  pretty_urls = false

# Only deploy main branch as a branch deploy.
# All other branches only get a Deploy Preview when a PR is opened.
[context.branch-deploy]
  command = "echo 'Branch deploy disabled — open a PR for a Deploy Preview'"

  for = "/_astro/*"
  [headers.values]

```

Key changes:
- `command = "npm run build"` -> `command = "pnpm run build"`
- Added `[context.branch-deploy]` block that overrides the build command with a no-op echo, preventing branch deploys from producing output
- Preserved `[build.processing.html]` and `[[headers]]` blocks unchanged

NOTE: The `netlify.toml` change alone is necessary but not sufficient for full FND-10 compliance. The Netlify UI must also be configured to restrict branch deploys. This is a one-time manual step documented in the checkpoint task below.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions to dist/ output | CI pipeline must validate build output before allowing merge |
| Netlify CONTEXT env var to HTML output | Build-time environment variable controls whether noindex meta is emitted |
| Netlify branch deploy to public URLs | Non-production deploy URLs must not be indexed by search engines |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Information Disclosure | dist/ build output | mitigate | Post-build grep gate in CI rejects any build containing banned AstroWind strings in HTML/JS/CSS files -- permanent enforcement on every PR |
| T-03-02 | Information Disclosure | Netlify Deploy Preview URLs | mitigate | Layout.astro emits noindex nofollow meta when CONTEXT is deploy-preview or branch-deploy -- prevents search engine indexing of preview URLs |
| T-03-03 | Spoofing | Netlify branch deploys | mitigate | netlify.toml context.branch-deploy overrides build command with no-op echo plus Netlify UI branch deploy policy set to production-only -- prevents unauthorized branch content from being publicly accessible |
| T-03-04 | Elevation of Privilege | .github/workflows/actions.yaml | accept | CI workflow runs on ubuntu-latest with default GITHUB_TOKEN permissions. No secrets are exposed to PR builds from forks (standard GitHub Actions security model). Risk accepted as low for a marketing site. |
</threat_model>
