## Component Patterns

### Case Study Cards (Index)

**Visual structure:**
- Container: Secondary color background (hsl(217 60% 93%)), padding lg (24px)
- Logo: Small (60x60px recommended), centered at top or left
- Title: 20px, 600 weight, Montserrat, dominant color
- Vertical badge: xs padding (4px), sm font size (14px), accent background
- Outcome snippet: 14px, 400 weight, PT Serif, muted color, 2-3 lines max
- Hover state: Subtle shadow lift (follows WidgetWrapper shadow pattern)

**Responsive:** 1 column mobile, 2 columns tablet, 3 columns desktop

### Blog Post List (Grid or List)

**Visual structure:**
- Container: Secondary color background, padding md (16px)
- Featured image (optional): Full-width, aspect ratio 16:9 or 4:3
- Title: 20px, 600 weight, Montserrat
- Meta: 14px, 400 weight, muted color (author, date, reading time)
- Excerpt: 16px, 400 weight, PT Serif, max 3 lines
- Tags: xs padding badges, secondary background, small font
- Read more link: "Mehr lesen" as tertiary button or text link

**Pagination:** 6-8 posts per page (reasonable scroll depth on mobile)

### Team Grid

**Visual structure:**
- Container: 2 columns mobile, 3 columns tablet, 4 columns desktop
- Photo: Square aspect ratio, centered, consistent crop (headshot or full-shoulder)
- Name: 18px, 600 weight, Montserrat, centered below photo
- Title: 14px, 400 weight, muted color, centered
- Bio: 14px, 400 weight, PT Serif, 2-3 lines, centered
- Spacing: md (16px) between cards

**Photo placeholders (dev):** Initials on light secondary background if photos unavailable (no silhouettes)

### Careers List

**Visual structure:**
- Each role: Card or list item with secondary background
- Role title: 20px, 600 weight, Montserrat
- Department tag: sm badge, accent background (10% color)
- Type badge: fulltime | internship | initiative (smaller badges)
- Description: 14px, 400 weight, PT Serif (optional, from content)
- Apply button: "Bewerben", btn-primary utility class, 44px min height

**Responsive:** Single column, full-width cards on all breakpoints

### Contact Page Split

**Visual structure:**
- Two-column layout (side-by-side) on desktop, stacked on mobile
- Each contact card: Photo (square, 200x200px minimum), name, role, phone (bold), email (link)
- Office address block: below contacts, two-column or full-width
- Maps container: Placeholder screenshot or fallback static map, "Karte laden" button at bottom
- Events section: Below maps, simple chronological list
- Final CTA: At page bottom, full-width button

**Accessibility:** All phone numbers are clickable `<a href="tel:...">` links.

## Notes for Executor

1. **Content Collections:** All Phase 6 pages read from existing `caseStudy`, `post`, `job`, `event`, `team` schemas in `src/content.config.ts`. No schema changes needed.

2. **Locale Routing:** Phase 6 follows i18n routing from Phase 3:
   - Case studies: `/fallstudien/` (DE) + `/en/case-studies/` (EN)
   - Blog: `/aktuelles/` (DE) + `/en/news/` (EN)
   - Team: `/team/` (DE) + `/en/team/` (EN)
   - Careers: `/karriere/` (DE) + `/en/careers/` (EN, shell redirect)
   - Contact: `/kontakt/` (DE) + `/en/contact/` (EN)

3. **Reusable Components:** Leverage existing widgets from Phase 4:
   - `CallToAction.astro` — every Phase 6 page ends with this
   - `Hero.astro` — case study detail hero with quote
   - `Content.astro` — case study problem/approach/outcome sections
   - `Brands.astro` — logo display for case study customer logos
   - Blog components (`Grid`, `Pagination`, `SinglePost`, `Tags`) — reuse as-is

4. **Brand Compliance:**
   - All CTAs use "Beratung anfragen" (DE) / "Book a consult" (EN)
   - All copy follows `voice.md` tone (preventive, no exclamation points, B2B)
   - Contact info pulled from `canonical.yaml` (single source of truth)

5. **Maps Integration:** Click-to-load iframe pattern for DSGVO compliance (CONT-02):
   - Initial state: Static screenshot or placeholder
   - Click "Karte laden": Replace with `<iframe>` from Google Maps
   - No iframe rendered until explicit user action

6. **Blog Pagination:** 6-8 posts per page to balance load time and scroll depth. Tag pages use same pagination.

7. **Events Auto-Hide:** Past events hidden by comparing `endDate` to current date at build time. No client-side filtering.
